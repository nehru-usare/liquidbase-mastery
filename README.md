# Liquibase Mastery: Spring Boot User Management Service

## 1. Project Overview

This is a production-style Spring Boot project that demonstrates database schema versioning and migrations using **Liquibase**. 
The application is a simple User Management REST API with basic CRUD operations. Instead of relying on Hibernate's `ddl-auto` to manage the database schema, we use Liquibase to track, manage, and apply schema changes reliably across different environments.

### Technologies Used
* Java 17+
* Spring Boot 3.2.x (Web, Data JPA)
* PostgreSQL
* Liquibase (YAML format)
* Docker & Docker Compose
* Maven
* Lombok

## 3. Maven pom.xml

Provides all the required spring-boot-starters, PostgreSQL driver, and the `liquibase-core` dependency. (See actual `pom.xml` in this repository).

## 4. application.yml

Configures the PostgreSQL datasource and the `spring.liquibase.change-log` property pointing to our master changelog file. Hibernate `ddl-auto` is set to `validate` to ensure Liquibase is the sole source of truth.

## 5. Liquibase Changelog Files

Liquibase tracks changes through "changelogs" and "changesets".

* **db.changelog-master.yaml**: The master file that simply includes all other migration files in the exact order they should be executed.
* **v1-create-users**: Creates the initial table.
* **v2-add-email**: Adds the email column.
* **v3-add-created-at**: Adds the timestamp column.
* **v4-add-index**: Adds an index to the email column.
* **v5-example-rollback**: Demonstrates defining a custom rollback statement.

## 11. Detailed Documentation & Liquibase Concepts

### What is Liquibase?
Liquibase is an open-source database-independent library for tracking, managing, and applying database schema changes.

### Why is Schema Versioning Important?
Just like application code is versioned using Git, the database schema also needs versioning. Direct manual updates or using Hibernate `update` in production can lead to lost data or out-of-sync environments. Schema versioning guarantees that all environments (Dev, QA, Prod) have identical database structures, allowing you to tie database changes directly to application releases.

### Liquibase Architecture
Liquibase runs on top of your application using JDBC. It uses two key tables automatically created in your target database:
* **DATABASECHANGELOG**: Tracks which changesets have been executed by storing the `id`, `author`, and `filename`. 
* **DATABASECHANGELOGLOCK**: Ensures only one instance of Liquibase runs at a time (crucial in distributed Microservices).

### How Liquibase Works on Startup
1. Spring Boot starts and initializes the Liquibase bean.
2. Liquibase locks the `DATABASECHANGELOGLOCK` table.
3. Liquibase reads `db.changelog-master.yaml` and parses the defined changesets sequentially.
4. For each changeset, it queries `DATABASECHANGELOG`. If the changeset ID/Author/File combination is not in the table, the changeset is executed, and a row is recorded.
5. Once complete, Liquibase releases the lock, and the Spring Boot application continues starting up.

### Changesets and Rollbacks
* **Changeset**: A uniquely identifiable chunk of database changes (e.g., creating a table).
* **Rollback**: Instructions on how to undo a changeset. Liquibase auto-generates rollbacks for many operations (like creating a table), but for destructive operations (like dropping a table) or complex changes, custom rollbacks must be provided (as seen in `v5-example-rollback.yaml`).

### Step-by-Step Instructions to Run
1. Make sure Docker is running on your machine.
2. Open terminal in the project root.
3. Run the following command:
   ```bash
   docker-compose up --build
   ```
4. The database and application will start up. Liquibase will automatically create the tables on boot.
5. The API will be available at `http://localhost:8080/api/users`.

## Example API Requests

**Create User**
```bash
curl -X POST http://localhost:8080/api/users \
-H "Content-Type: application/json" \
-d '{"firstName": "John", "lastName": "Doe", "email": "john.doe@example.com"}'
```

**Get All Users**
```bash
curl http://localhost:8080/api/users
```

**Get User by ID**
```bash
curl http://localhost:8080/api/users/1
```

**Update User**
```bash
curl -X PUT http://localhost:8080/api/users/1 \
-H "Content-Type: application/json" \
-d '{"firstName": "Johnny", "lastName": "Doe", "email": "johnny.updated@example.com"}'
```

**Delete User**
```bash
curl -X DELETE http://localhost:8080/api/users/1
```

## Migration Workflow Explanation

If a developer wants to modify the database schema:
1. **Create a new changeset file**: E.g., `v6-add-phone-number.yaml`.
2. **Include in Master Changelog**: Open `db.changelog-master.yaml` and append `- include: file: db/changelog/changesets/v6-add-phone-number.yaml`.
3. **Deploy the application**: When the Spring Boot app restarts or is deployed to the server, Liquibase automatically detects the new file, applies it to the DB, and records the execution in `DATABASECHANGELOG`.

## Advanced Section

### Best Practices in Microservices
* **One Database per Microservice**: Each microservice must own its schema. Do not share databases across domains.
* **Separation of Scripts**: Keep changelogs modular and versioned properly (e.g., grouped by releases `1.0.0`, `1.1.0`).
* **Backward Compatible Changes**: Always deploy database changes that are compatible with the *current* running application. For drops or renames, use a multi-phase deployment (Phase 1: Add new column; Phase 2: Start using new column; Phase 3: Drop old column).

### CI/CD Integration
* Liquibase should run as part of the CD pipeline. Often, it's run via the Liquibase Maven/Gradle plugin or CLI against the target database *before* the application container is replaced, ensuring the DB is ready for the new code.
* `updateTestingRollback` CLI command can be used in CI pipelines to guarantee a migration script can be cleanly applied and rolled back before merging to the main branch.

### Liquibase vs Flyway
* **Flyway**: Simpler, utilizes raw SQL scripts. Great if you primarily write raw SQL and target a single database vendor.
* **Liquibase**: Can use XML, YAML, JSON formats. It abstracts the SQL, meaning migrations can be run against PostgreSQL, MySQL, or Oracle without rewriting them. It also provides advanced features like rollbacks and database diffing. Choose Liquibase if you need complex migrations, multiple DB support, or standardized structuring via YAML/XML.
